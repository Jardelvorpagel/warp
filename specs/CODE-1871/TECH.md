# Up-arrow prompt history in the TUI — implementation plan (CODE-1871)
See [`PRODUCT.md`](./PRODUCT.md) for user-facing behavior. This plan targets `crates/warp_tui` and adds a thin `tui_export` seam over the GUI's existing prompt-history data. References are pinned to commit `7ffccbb8223e3e66ef5b850a85a546fd3c3ab7e7`.
## Context
The TUI already has a complete, reusable inline-menu framework that the prompt-history menu should slot into as one more menu, alongside slash commands, `/conversations`, model, skills, and MCP.

### Reusable-vs-TUI-specific decision
- **Shared (reuse via `tui_export`): the data fetch.** The prompt list must come from `BlocklistAIHistoryModel::all_ai_queries` so the TUI and GUI read the same history. This is the "share code paths with the GUI" requirement. `all_ai_queries` is `pub(crate)`; expose it (and `AIQueryHistory`) through `app/src/tui_export.rs`. The de-dupe + ordering logic in `sort_and_dedupe_suggestions` and the prefix-filter should also be shared rather than reimplemented — see the helper option below.
- **TUI-specific (new): the menu model, the open trigger, the first-visual-row check, and buffer snapshot/restore.** The GUI's `InlineHistoryMenuView`, `SearchMixer`, `InputSuggestionsModeModel`, and `single_cursor_on_first_row` are all GUI-view/GUI-framework constructs and are not reusable in the headless TUI. The TUI already has parallel primitives (`TuiInlineMenuListState`, `TuiInputSuggestionsModeModel`, the char-cell `DisplayLattice`) that the new menu model builds on.
## Proposed changes
### 1. Expose the prompt-history fetch to the TUI (`app` crate)
- In `app/src/ai/blocklist/history_model.rs`, make `all_ai_queries` callable from `warp_tui`. **Decision (confirmed at review): option (a), the small shared helper.**
  - **(a) Add a small shared helper** (e.g. `pub fn tui_prompt_history(terminal_surface_id: EntityId, query: &str, app: &AppContext) -> Vec<AIQueryHistory>`) next to `up_arrow.rs` that runs `all_ai_queries(Some(id))`, applies the same prefix filter and `sort_and_dedupe` used by the GUI agent-view path, and returns owned rows. This keeps the ordering/dedupe/filter logic in one place and gives the TUI a simple, testable entry point. Re-export it from `tui_export.rs`.
  - **(b)** Alternatively just re-export `all_ai_queries` + `AIQueryHistory` and reimplement filter/dedupe in the TUI. Rejected: duplicates GUI ordering/dedupe semantics (invariants 8–9) and risks drift.
- Add the re-export(s) in [`app/src/tui_export.rs`](https://github.com/warpdotdev/warp/blob/7ffccbb8223e3e66ef5b850a85a546fd3c3ab7e7/app/src/tui_export.rs) under the existing `history_model` block (line 50).
### 2. First-visual-row check for the char-cell editor
The TUI cannot use `single_cursor_on_first_row` (GUI view). Add an equivalent on the char-cell path:
- Add a helper on `TuiInputView` (or `CodeEditorModel`/`CharCellState`) that returns whether the single caret is on the first visual row. Implement via the char-cell display lattice: project the cursor offset with `DisplayLattice::offset_to_display_point(cursor_offset)` ([`crates/editor/src/render/model/char_cell_display.rs` (139-195) @ 7ffccbb](https://github.com/warpdotdev/warp/blob/7ffccbb8223e3e66ef5b850a85a546fd3c3ab7e7/crates/editor/src/render/model/char_cell_display.rs#L139-L195)) and check `row == 0`. `TuiInputView::follow_cursor` already shows the access pattern for reaching `char_cell()` from the render state ([`input/view.rs` (1079-1088)](https://github.com/warpdotdev/warp/blob/7ffccbb8223e3e66ef5b850a85a546fd3c3ab7e7/crates/warp_tui/src/input/view.rs#L1079-L1088)). Require a single caret with no active selection (reuse the `selection_range(ctx).is_none()` idiom from `is_cursor_at_start`).
### 3. New menu model: `TuiPromptHistoryMenuModel`
New file `crates/warp_tui/src/prompt_history_menu.rs` (+ `prompt_history_menu_tests.rs`), modeled on `conversation_menu.rs`:
- State enum `Closed` / `Open { list: TuiInlineMenuListState<TuiPromptHistoryRow>, original_buffer: String }`. `TuiPromptHistoryRow { text: String }`.
- `open(ctx)`: `suggestions_mode.try_open(TuiInputSuggestionsMode::PromptHistory, ctx)`; **snapshot the current input text into `original_buffer`** (do NOT clear the buffer — unlike the conversation menu — because the typed text is the initial query); `refresh_rows`.
- Subscribe to the input editor `ContentChanged` (like conversation menu) to `refresh_rows` while open — but guard against the preview writes re-querying (see below). Subscribe to `BlocklistAIHistoryModel` events to `refresh_rows` while open (invariant 27).
- `refresh_rows(ctx)`: read the **active query** (see preview note), call the shared `tui_prompt_history(terminal_surface_id, &query, ctx)`, map to rows, `list.replace_rows(...)` with stable-selection reconcile and `preferred_index = last row` on first populate (invariant 10). Empty → `TuiInlineMenuStatus::Empty("No prompt history"/"No matching prompts")`.
- `select_previous`/`select_next`: delegate to `list`, then **preview** the selected row into the input buffer (invariant 14). Preview uses an undo-agnostic set (mirror GUI `set_buffer_text_ignoring_undo`; on the TUI use `clear_buffer` + `user_insert` or an ignoring-undo setter — confirm the `CodeEditorModel` API). To keep query stable while previewing (invariant 14), store the user's typed query on the `Open` state (`query: String`) and treat editor `ContentChanged` as a query change **only when it did not originate from a preview write** (set a `previewing` guard flag around preview writes, matching the GUI reading query from the mixer rather than the buffer).
- `accept_selected(ctx) -> Option<String>`: return the selected row's `text` (the session view submits it — see wiring).
- `dismiss(ctx)`: restore the input buffer to `original_buffer` (invariant 18), then `close_if_active(PromptHistory)`.
- `snapshot(ctx)`: `TuiInlineMenuSnapshot` with header title "Prompt history", rows (`TuiInlineMenuRowStyle::Default`, `is_selectable: true`), selection/scroll, and empty/loading status. `MAX_VISIBLE_ROWS = result_row_capacity(MAX_INLINE_MENU_ROWS, true, false)` (title, no tabs) — same as conversation menu.
### 4. Suggestions mode + menu registration
- Add `PromptHistory` variant to `TuiInputSuggestionsMode` and handle it in the exhaustive `try_open` match ([`input_suggestions_mode.rs` (70-75)](https://github.com/warpdotdev/warp/blob/7ffccbb8223e3e66ef5b850a85a546fd3c3ab7e7/crates/warp_tui/src/input_suggestions_mode.rs#L70-L75)).
- Implement `TuiInlineMenuHandle for ModelHandle<TuiPromptHistoryMenuModel>` in `inline_menu.rs` (mode = `PromptHistory`; `accept` maps the returned `String` to a new `TuiInlineMenuAccepted::PromptHistory(String)` variant). Update the exhaustive `TuiInlineMenuAccepted` match in `input/view.rs` `handle_inline_menu_action` and add a `TuiInputViewEvent::AcceptedPromptHistory(String)`.
- In `terminal_session_view.rs`: construct the model, push `TuiInlineMenu::new(prompt_history_menu.clone())` into `inline_menus`, store the handle on the view, subscribe for `ctx.notify()`, and handle `AcceptedPromptHistory(text)` → set input text and submit via the existing send-prompt path (mirrors `handle_accepted_*`). Add the field to the struct literal (lines 941-971).
### 5. Open trigger in `TuiInputView`
- In `handle_action`, before the fall-through `MoveUp => m.move_up`, add: if no active inline menu, not shell mode, and the new first-visual-row check passes, call `prompt_history_menu.open(ctx)` and return. This is the single new branch; menu-open cycling (invariant 3) is already handled by `handle_inline_menu_action`. Down-past-last closing (invariant 20) is optional parity — if included, add a close+restore branch to the menu's `select_next` when already on the last row.
### 6. No feature flag
- Per review decision, this ships **without** a feature flag: the open trigger in change (5) is not gated. No `FeatureFlag` variant is added, and invariant 6 is satisfied by shipping the behavior enabled by default.
## Testing and validation
Primary verification is render-to-lines unit tests per the `tui-testing` skill (element/screen tests in `*_tests.rs`), since the TUI has no GUI-style integration harness.
- **Model tests** (`prompt_history_menu_tests.rs`, using `App::test`): open populates rows from a mocked `BlocklistAIHistoryModel`; ordering/dedupe (invariants 8–9); default selection is the last row (10); prefix filter (11); stable selection across refresh (12, 27); whitespace prompts excluded (13); `dismiss` restores `original_buffer` (18, 22, 23); `accept_selected` returns the selected text (16); empty-history status (22).
- **Input-view tests** (`input/view_tests.rs`): Up on first visual row opens the menu (1); Up on a lower row still moves the cursor and does not open (2); Up while another menu is open cycles that menu, not prompt history (3); shell mode suppresses opening (25); Escape closes + restores (18–19); preview-on-select updates the buffer without changing the query (14); Up opens the menu unconditionally with no flag gating it (6). Use the `render_lines` / `dispatch_event` helpers from `transcript_view_tests.rs`.
- **Session-view test** (`terminal_session_view_tests.rs`): `AcceptedPromptHistory` submits the selected prompt through the send-prompt path (16).
- **Shared helper test** (`app` crate): `tui_prompt_history` applies the same filter/dedupe/order as the GUI agent-view data source (guards invariants 8–9, 11 against drift).
- **Manual**: run `CARGO_BUILD_JOBS=1 ./script/run-tui` and verify Up on the top line opens the menu, typing filters, arrowing previews, Enter accepts, Escape restores — per the `tui-verify-change` skill.
- Before PR: `./script/format` and `cargo clippy` (per `AGENTS.md`), and `cargo nextest run -p warp_tui -p warpui_core`.
## Risks and mitigations
- **Buffer dual-use (query vs. preview).** The input buffer is both the search query and the preview surface. Mitigation: store the typed query on the `Open` state and gate re-query on a `previewing` flag so preview writes don't clobber the query — this is exactly how the GUI separates them (query read from the mixer, not the buffer). This is the trickiest part; cover it with the invariant-14 test.
- **`all_ai_queries` visibility / export churn.** Prefer the shared-helper option (1a) to keep GUI ordering/dedupe authoritative and minimize surface exposed to `warp_tui`.
- **Escape / Down priority regressions.** New branches must not disturb the existing Escape-dismisses-menu-first ordering or the `handle_inline_menu_action` gating; keep the open trigger as a single guarded branch and lean on existing exhaustive matches to catch missed sites.
- **`is_child_agent_conversation` / ambient sessions.** `all_ai_queries` already excludes ambient sessions and the synthetic orchestrator prompt; reusing it (vs. a bespoke TUI fetch) inherits those exclusions for free.
## Follow-ups
- Optional parity extensions later: command/conversation tabs (out of scope here), and honoring the GUI's `IgnoredSuggestionsModel` if prompt-ignoring is wanted in the TUI.
## Parallelization
Not proposed. The change is a single, tightly-coupled vertical slice in one crate (`warp_tui`) plus a small `tui_export` seam; the menu model, open trigger, and wiring share types and must land together, so splitting across agents would add coordination cost without reducing wall-clock time. A single implementer on branch `code-1871-up-arrow-prompt-history` should carry it end to end.
